# 4. ログデータの確認とテーブルモード

この研修では、前回のマニュアル操作で保存した走行ログ（Log）がどのようなものかを確認し、そのログデータを利用して走行を再現する「テーブルモード」について学びます。

**学習目標:**
- `DATA_SAVE=1`で保存されるログデータのフォルダ構成と内容を理解する
- `metadata.csv`に含まれる詳細な走行データの意味を知る
- テーブルモード（`MODE_NUM=2`）を使い、CSVファイルに基づいてロボットを走行させる方法を習得する
- ログデータを編集し、独自の走行パターンを作成する基礎を学ぶ

---

## 1. ログデータの確認

`03_Manual_Control`の研修で`DATA_SAVE=1`に設定して走行すると、`Robot1/training_data/`フォルダ内に`run_[日付]_[時刻]`という名前のフォルダが作成されます。これが1回の走行に関する全てのデータが詰まった「ログフォルダ」です。

### 1.1 ログフォルダの構成
ログフォルダの中には、主に以下のファイルとフォルダが格納されています。

- **`images/`フォルダ**
  - 走行中にロボットのカメラが捉えた全フレームの画像（JPEG形式）が保存されています。

- **`metadata.csv`**
  - その走行に関する**最も重要なデータファイル**です。走行中のロボットの全状態が、1tick（シミュレーションの1コマ）ごとに記録されています。

- **`output_video.mp4`**
  - `images/`フォルダの画像を連結して自動生成された、走行のリプレイ動画です。

- `UnityLog.txt`, `terminal_log.txt`
  - デバッグ用の詳細な動作ログです。
  - Unity側、およびPython側で処理された動作ログが保存されています。

---


### 1.2 `metadata.csv` の中身
`metadata.csv`をExcelやGoogleスプレッドシートなどの表計算ソフトで開いてみましょう。以下の **16個のカラム（列）** が左から順番に並んでいます。

#### 時系列・識別情報

| カラム名 | 内容 |
|----------|------|
| **`id`** | Tickの連番インデックス。ファイル全体を通して一意な番号です。 |
| **`session_time_ms`** | スタートシーケンス開始からの経過時間（ミリ秒）。カウントダウン中も計測されます。 |
| **`race_time_ms`** | 「GO」シグナルが発火した瞬間からの経過時間（ミリ秒）。カウントダウン中は `0` のままです。 |
| **`filename`** | `images/`フォルダ内の対応する画像ファイル名（例: `frame_000079.jpg`）。 |

#### ロボットの操作・状態

| カラム名 | 内容 |
|----------|------|
| **`soc`** | バッテリー残量（充電率）。`1.0` が満充電、`0.0` が空。 |
| **`drive_torque`** | 駆動輪トルクの正規化値。`+1.0` が最大前進、`-1.0` が最大後進、`0.0` が停止。 |
| **`steer_angle`** | 前輪の操舵角（**ラジアン単位**）。最大 **±0.524 rad（±30度）** の範囲。負の値（`-`）が**左**、正の値（`+`）が**右**。 |
| **`status`** | ロボットの状態を表す文字列（下表参照）。 |

**`status` の値一覧:**

| 値 | 意味 |
|----|------|
| `StartSequence` | スタートのカウントダウン中 |
| `Running` | 走行中 |
| `Lap0` | ラップ計測開始地点（学習時に使用） |
| `Lap1` / `Lap2` ... | ラップ通過後 |
| `Finish` | 規定周回を完走 |
| `FalseStart` | フライング（GO前に動いた）→ 失格 |
| `Fallen` | コース外に落下 → 退場 |
| `BatteryDepleted` | バッテリー切れ → 走行不能（障害物化） |
| `ForceEnd` | 強制終了 |

#### 位置・姿勢

> **注意**: 列の並び順は `pos_z, pos_x, yaw, pos_y` です。アルファベット順ではありません。  
> **[重要]** `steer_angle` は **ラジアン単位** ですが、`yaw` は **度単位** です。データ分析の際はご注意ください。

| カラム名 | 内容 |
|----------|------|
| **`pos_z`** | Z座標 [m]。**前後方向**の位置（コース進行方向が正）。 |
| **`pos_x`** | X座標 [m]。**左右方向**の位置（右が正）。 |
| **`yaw`** | ヨー角（進行方向）[**度**]。前進方向=0、右回りが正の値。 |
| **`pos_y`** | Y座標 [m]。**上下方向**の位置（上が正）。`-0.1m` 未満でコース外落下と判定。 |

#### エラー・衝突情報

| カラム名 | 内容 |
|----------|------|
| **`error_code`** | エラー状態を示す数値コード。通常走行時は `999`。 |
| **`collision_type`** | そのTickの衝突種別: `"wall"`（壁）、`"robot"`（他ロボット）、`"both"`（両方）、`""`（衝突なし）。 |
| **`collision_penalty`** | そのTickに適用された衝突ペナルティ率（通常は `0.0`）。バッテリーから即時減算される。 |
| **`collision_target`** | 衝突した相手の名前（例: `Robot2`、`Wall`）。衝突なしの場合は空。 |

このファイルを見ることで、「どの瞬間に、どんな操作をして、その結果ロボットがどこにいたか、衝突はあったか」を詳細に分析することができます。

## 2. テーブルモードでの走行再現

テーブルモード（`MODE_NUM=2`）は、`Robot1/table_input.csv`というファイルに記述された`drive_torque`と`steer_angle`の値を上から順番に読み込み、その通りにロボットを動かすモードです。
つまり、**走行の設計図（テーブル）に基づいて、ロボットが自動で走ります。**

### `table_input.csv`の形式
このファイルは非常にシンプルで、3つのカラムしかありません。
- `time_id`: 動作の通し番号（0から始まる整数）。
- `drive_torque`: アクセル操作。
- `steer_angle`: ハンドル操作。

## 3. 研修課題

### 課題1: ログデータから走行を再現する

1.  **事前準備**
    - `config.txt`の`ACTIVE_ROBOTS`を`1`に設定します。
    - `Robot1/robot_config.txt`を以下のように編集します。
      ```ini
      
      MODE_NUM=2
      

      ```

2.  **テーブルデータの作成**
    - 前回の練習で作成されたログフォルダを開き、`metadata.csv`を表計算ソフトで開きます。
    - `drive_torque`と`steer_angle`の2つの列のデータを、ゴールするまでの行（例: 1000行目までなど）選択してコピーします。

3.  **`table_input.csv`の編集**
    - `Robot1/table_input.csv`をテキストエディタや表計算ソフトで開きます。
    - **A列(`time_id`)以外の、`drive_torque`, `steer_angle`の古いデータを全て削除**します。
    - 先ほどコピーした`drive_torque`, `steer_angle`のデータをB列、C列に貼り付けます。
    - **A列(`time_id`)を、`0`から始まる連番で埋め直します。** Excelなどを使う場合は、`0`、`1`と入力してフィルハンドルをドラッグすると簡単です。
    - 編集が終わったら、ファイルをCSV形式で保存します。

4.  **走行の確認**
    - `main.py`を実行します。
    - ロボットが、先ほどコピーしたログ通りに自動で走行を始めることを確認しましょう。自分のキーボード操作が再現される様子は、少し不思議な感覚かもしれません。

> **考えてみよう: 完全に同じ動きになりましたか？**
>
> きれいに周回できた人は少ないのではないでしょうか。これは**「遅れ（レイテンシ）」**が原因です。
>
> 皆さんがログとして使ったデータは、ロボットが**実際に受信した**操作値です。そして今回のテーブルモードでは、そのデータを上から順番に **20fps（50ms周期）** で送信しました。ロボット側はそれを受け取って動きます。
>
> しかし、実際にはいくつかの「ズレ」が生じます。
> - **通信の遅れ**: 送信が必ず成功するとは限らず、パケットの到着タイミングにばらつきがある
> - **機械的な遅れ**: トルクを指示してから実際にタイヤが回転するまでには、わずかな時間差がある
>
> これは現実の世界でも同じです。「決められた指令通りにただ動く」だけでは、外からの影響（外乱）には対応できません。このような制御方式を**シーケンス制御**と呼びます。
>
> 思い通りに動かすためには、余裕を持った指令値の工夫、またはこの後学ぶ**フィードバック制御**が必要になります。

### 課題2: 走行データを編集してみる（応用）

- `table_input.csv`の特定区間の`steer_angle`を全て`0`に書き換えて保存し、ロボットがどのように動くか観察してみましょう（直線的に走るはずです）。
- 最初のカーブだけ、`drive_torque`の値を半分に書き換えて、減速して曲がるように編集してみましょう。

このように、テーブルモードはAIや複雑なプログラムなしに、精密な自動運転パターンを作成・テストするのに非常に有効です。

### 課題3: `metadata.csv` を可視化してみよう

ログデータは数字の羅列ですが、**グラフにすることで走行の様子が「見える」**ようになります。
ExcelやGoogleスプレッドシート、あるいはPython（`pandas` + `matplotlib`）など、使いやすいツールで試してみましょう。

---

#### 課題3-1: 走行ルートを描く（散布図）

`pos_z`（横軸）と `pos_x`（縦軸）を散布図でプロットしてみましょう。

> **ヒント**: X軸に `pos_z`、Y軸に `pos_x` を設定し、データ点を線でつなぐと走行軌跡になります。

- コースのどんな形が見えますか？
- スタート地点とゴール地点はどのあたりですか？
- 周回ごとに軌跡がずれている箇所はありますか？

---

#### 課題3-2: 操作の時系列を見る（折れ線グラフ）

横軸を `race_time_ms`、縦軸を `drive_torque` と `steer_angle` として折れ線グラフを描いてみましょう（2本の線を重ねて描くと比較しやすいです）。

> **ヒント**: `race_time_ms` が `0` の行はカウントダウン中です。値が `0` より大きい部分だけを使うと、実際の走行区間が分かりやすくなります。

- アクセルを踏み込んでいる区間と、緩めている区間はどこですか？
- ハンドルを大きく切っている瞬間は、`drive_torque` とどんな関係がありますか？
- カーブと直線で、操作のパターンに違いは見えますか？

---

#### 課題3-3: 位置と向きの時系列を見る（折れ線グラフ）

横軸を `race_time_ms` として、`pos_z`・`pos_x`・`yaw` の3本を時系列グラフで描いてみましょう。

- `pos_z` と `pos_x` が大きく変化するタイミングは一致していますか？それとも交互に変化しますか？（それは何を意味しますか？）
- `yaw` が急激に変化している区間は、散布図（課題3-1）のどの部分に対応していますか？
- `yaw` の変化と、課題3-2の `steer_angle` の変化を見比べてみましょう。どんな関係が見えますか？
